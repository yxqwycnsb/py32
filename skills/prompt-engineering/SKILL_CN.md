# 提示工程模式

最大化 LLM 性能、可靠性和可控性的高级提示工程技术。

## 核心能力

### 1. 少量学习

通过展示示例而不是解释规则来教导模型。包括 2-5 个输入-输出对，演示所需行为。当您需要一致格式、特定推理模式或处理边缘情况时使用。更多示例提高准确性但消耗令牌——基于任务复杂性平衡。

**示例：**

```markdown
从支持票据中提取关键信息：

输入："我的登录不起作用，一直得到错误 403"
输出：{"issue": "authentication", "error_code": "403", "priority": "high"}

输入："功能请求：在设置中添加暗模式"
输出：{"issue": "feature_request", "error_code": null, "priority": "low"}

现在处理："无法上传超过 10MB 的文件，得到超时"
```

### 2. 思维链提示

在最终答案前请求逐步推理。添加"让我们逐步思考"（零样本）或包括示例推理跟踪（少样本）。用于需要多步逻辑的复杂问题、数学推理，或当您需要验证模型思维过程时。通过 30-50% 提高分析任务的准确性。

**示例：**

```markdown
分析这个错误报告并确定根本原因。

逐步思考：
1. 预期行为是什么？
2. 实际行为是什么？
3. 最近什么变化可能导致这个？
4. 涉及哪些组件？
5. 最可能的根本原因是什么？

错误："用户在昨天部署缓存更新后无法保存草稿"
```

### 3. 提示优化

通过测试和完善系统地改进提示。从简单开始，测量性能（准确性、一致性、令牌使用），然后迭代。对多样输入进行测试，包括边缘情况。使用 A/B 测试比较变体。对一致性和成本重要的生产提示至关重要。

**示例：**

```markdown
版本 1（简单）："总结这篇文章"
→ 结果：长度不一致，错过关键点

版本 2（添加约束）："用 3 个要点总结"
→ 结果：更好结构，但仍错过细微差别

版本 3（添加推理）："识别 3 个主要发现，然后总结每个"
→ 结果：一致、准确、捕捉关键信息
```

### 4. 模板系统

使用变量、条件部分和模块化组件构建可重用提示结构。用于多轮对话、基于角色的交互，或当相同模式适用于不同输入时。减少重复并确保类似任务的一致性。

**示例：**

```python
# 可重用代码审查模板
template = """
审查这个 {language} 代码的 {focus_area}。

代码：
{code_block}

提供反馈：
{checklist}
"""

# 使用
prompt = template.format(
    language="Python",
    focus_area="安全漏洞",
    code_block=user_code,
    checklist="1. SQL 注入\n2. XSS 风险\n3. 认证"
)
```

### 5. 系统提示设计

设置在对话中持续的全局行为和约束。定义模型角色、专业水平、输出格式和安全指南。用于不应在轮次间变化的稳定指令，释放用户消息令牌用于可变内容。

**示例：**

```markdown
系统：您是专门从事 API 设计的资深后端工程师。

规则：
- 始终考虑可扩展性和性能
- 默认建议 RESTful 模式
- 立即标记安全问题
- 用 Python 提供代码示例
- 使用早期返回模式

格式响应为：
1. 分析
2. 推荐
3. 代码示例
4. 权衡
```

## 关键模式

### 渐进披露

从简单提示开始，仅在需要时添加复杂性：

1. **级别 1**：直接指令
   - "总结这篇文章"

2. **级别 2**：添加约束
   - "用 3 个要点总结这篇文章，重点关注关键发现"

3. **级别 3**：添加推理
   - "阅读这篇文章，识别主要发现，然后用 3 个要点总结"

4. **级别 4**：添加示例
   - 包括 2-3 个带有输入-输出对的示例总结

### 指令层次

```
[系统上下文] → [任务指令] → [示例] → [输入数据] → [输出格式]
```

### 错误恢复

构建优雅处理失败的提示：

- 包括后备指令
- 请求置信度分数
- 当不确定时请求替代解释
- 指定如何指示缺失信息

## 最佳实践

1. **具体**：模糊提示产生不一致结果
2. **展示，不要讲述**：示例比描述更有效
3. **广泛测试**：在多样、代表性输入上评估
4. **快速迭代**：小变化可以有大影响
5. **监控性能**：在生产中跟踪指标
6. **版本控制**：将提示视为具有适当版本控制的代码
7. **记录意图**：解释为什么提示结构如其所是

## 常见陷阱

- **过度工程**：在尝试简单提示前从复杂提示开始
- **示例污染**：使用与目标任务不匹配的示例
- **上下文溢出**：用过多示例超过令牌限制
- **模糊指令**：留下多个解释的空间
- **忽略边缘情况**：不在异常或边界输入上测试

## 集成模式

### 与 RAG 系统结合

```python
# 将检索上下文与提示工程结合
prompt = f"""给定以下上下文：
{retrieved_context}

{few_shot_examples}

问题：{user_question}

仅基于上述上下文提供详细答案。如果上下文不包含足够信息，明确说明缺少什么。"""
```

### 与验证结合

```python
# 添加自我验证步骤
prompt = f"""{main_task_prompt}

生成响应后，验证它是否满足这些标准：
1. 直接回答问题
2. 仅使用提供的上下文中的信息
3. 引用特定来源
4. 承认任何不确定性

如果验证失败，修改您的响应。"""
```

## 性能优化

### 令牌效率

- 移除冗余词和短语
- 在首次定义后一致使用缩写
- 整合相似指令
- 将稳定内容移动到系统提示

### 延迟减少

- 在不牺牲质量的情况下最小化提示长度
- 对长格式输出使用流式传输
- 缓存常见提示前缀
- 在可能时批量类似请求

# 代理提示最佳实践

基于 Anthropic 的代理提示官方最佳实践。

## 核心原则

### 上下文窗口

"上下文窗口"是指语言模型在生成新文本时可以回顾和引用的文本总量加上它生成的新文本。这不同于模型训练的大型语料库，而是代表模型的"工作记忆"。更大的上下文窗口允许模型理解和响应更复杂、更长的提示，而较小的上下文窗口可能限制模型处理长提示或在扩展对话中保持连贯性的能力。

- 渐进令牌积累：随着对话通过轮次推进，每个用户消息和助手响应在上下文窗口中积累。之前的轮次完全保留。
- 线性增长模式：上下文使用随每个轮次线性增长，之前的轮次完全保留。
- 200K 令牌容量：总可用上下文窗口（200,000 令牌）代表存储对话历史和从 Claude 生成新输出的最大容量。
- 输入-输出流：每个轮次包括：
  - 输入阶段：包含所有之前的对话历史加上当前用户消息
  - 输出阶段：生成成为未来输入一部分的文本响应

### 简洁是关键

上下文窗口是公共资源。您的提示、命令、技能与 Claude 需要知道的一切共享上下文窗口，包括：

- 系统提示
- 对话历史
- 其他命令、技能、钩子、元数据
- 您的实际请求

**默认假设**：Claude 已经非常聪明

只添加 Claude 没有的上下文。挑战每条信息：

- "Claude 真的需要这个解释吗？"
- "我可以假设 Claude 知道这个吗？"
- "这个段落是否值得其令牌成本？"

**好示例：简洁**（大约 50 个令牌）：

````markdown  theme={null}
## 提取 PDF 文本

使用 pdfplumber 进行文本提取：

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

**坏示例：太冗长**（大约 150 个令牌）：

```markdown  theme={null}
## 提取 PDF 文本

PDF（便携文档格式）文件是一种常见文件格式，包含文本、图像和其他内容。要从 PDF 中提取文本，您需要使用库。有许多可用于 PDF 处理的库，但我们推荐 pdfplumber，因为它易于使用并处理大多数情况良好。首先，您需要使用 pip 安装它。然后您可以使用下面的代码...
```

简洁版本假设 Claude 知道 PDF 是什么以及库如何工作。

### 设置适当的自由度

将具体程度与任务的脆弱性和可变性匹配。

**高自由度**（基于文本的指令）：

使用当：

- 多个方法有效
- 决策取决于上下文
- 启发式指导方法

示例：

```markdown  theme={null}
## 代码审查过程

1. 分析代码结构和组织
2. 检查潜在错误或边缘情况
3. 建议可读性和可维护性改进
4. 验证项目约定遵守
```

**中等自由度**（带有参数的伪代码或脚本）：

使用当：

- 存在首选模式
- 一些变体可接受
- 配置影响行为

示例：

````markdown  theme={null}
## 生成报告

使用此模板并根据需要自定义：

```python
def generate_report(data, format="markdown", include_charts=True):
    # 处理数据
    # 以指定格式生成输出
    # 可选包含可视化
```
````

**低自由度**（特定脚本，少或无参数）：

使用当：

- 操作脆弱且容易出错
- 一致性至关重要
- 必须遵循特定序列

示例：

````markdown  theme={null}
## 数据库迁移

运行完全这个脚本：

```bash
python scripts/migrate.py --verify --backup
```

不要修改命令或添加额外标志。
````

**类比**：将 Claude 视为探索路径的机器人：

- **两侧有悬崖的窄桥**：只有一种安全前进方式。提供特定护栏和确切指令（低自由度）。示例：必须按确切序列运行的数据库迁移。
- **没有危险的开阔田野**：许多路径通向成功。给出一般方向并信任 Claude 找到最佳路线（高自由度）。示例：上下文决定最佳方法的代码审查。

# 代理通信说服原则

用于编写提示，包括但不限于：Claude Code 的命令、钩子、技能，或子代理或任何其他 LLM 交互的提示。

## 概述

LLM 响应与人类相同的说服原则。理解这个心理学帮助您设计更有效的技能 - 不是为了操纵，而是确保即使在压力下也遵循关键实践。

**研究基础：**Meincke 等 (2025) 用 N=28,000 AI 对话测试了 7 个说服原则。说服技术将合规率提高了一倍以上（33% → 72%，p < .001）。

## 七个原则

### 1. 权威

**它是什么：** 对专业知识、凭证或官方来源的尊重。

**它如何在提示中工作：**

- 命令式语言："YOU MUST", "Never", "Always"
- 非协商框架："No exceptions"
- 消除决策疲劳和合理化

**何时使用：**

- 纪律执行技能（TDD、验证要求）
- 安全关键实践
- 既定最佳实践

**示例：**

```markdown
✅ 先写测试再写代码？删除它。从头开始。没有例外。
❌ 考虑在可行时先写测试。
```

### 2. 承诺

**它是什么：** 与先前行动、声明或公开声明的一致性。

**它如何在提示中工作：**

- 要求公告："Announce skill usage"
- 强制明确选择："Choose A, B, or C"
- 使用跟踪：TodoWrite 用于检查清单

**何时使用：**

- 确保技能实际被遵循
- 多步过程
- 问责机制

**示例：**

```markdown
✅ 当您找到技能时，您 MUST 公告："I'm using [Skill Name]"
❌ 考虑让您的伙伴知道您正在使用哪个技能。
```

### 3. 稀缺性

**它是什么：** 来自时间限制或有限可用性的紧迫性。

**它如何在提示中工作：**

- 时间绑定要求："Before proceeding"
- 顺序依赖："Immediately after X"
- 防止拖延

**何时使用：**

- 立即验证要求
- 时间敏感工作流
- 防止"我会稍后做"

**示例：**

```markdown
✅ 完成任务后，IMMEDIATELY 请求代码审查再继续。
❌ 您可以在方便时审查代码。
```

### 4. 社会证明

**它是什么：** 符合他人所做或被视为正常的事情。

**它如何在提示中工作：**

- 普遍模式："Every time", "Always"
- 失败模式："X without Y = failure"
- 建立规范

**何时使用：**

- 记录普遍实践
- 警告常见失败
- 强化标准

**示例：**

```markdown
✅ 没有 TodoWrite 跟踪的检查清单 = 步骤被跳过。每一次。
❌ 有些人发现 TodoWrite 对检查清单有帮助。
```

### 5. 团结

**它是什么：** 共享身份、"我们-ness"，内部群体归属感。

**它如何在提示中工作：**

- 协作语言："our codebase", "we're colleagues"
- 共享目标："we both want quality"

**何时使用：**

- 协作工作流
- 建立团队文化
- 非等级实践

**示例：**

```markdown
✅ 我们是同事一起工作。我需要您诚实的专业判断。
❌ 您应该告诉我如果我错了。
```

### 6. 互惠

**它是什么：** 回报收到利益的义务。

**它如何工作：**

- 谨慎使用 - 可能感觉操纵性
- 很少在提示中需要

**何时避免：**

- 几乎总是（其他原则更有效）

### 7. 喜欢

**它是什么：** 偏好与我们喜欢的人合作。

**它如何工作：**

- **不要用于合规**
- 与诚实反馈文化冲突
- 创建阿谀奉承

**何时避免：**

- 总是用于纪律执行

## 按提示类型组合原则

| 提示类型 | 使用 | 避免 |
|------------|-----|-------|
| 纪律执行 | 权威 + 承诺 + 社会证明 | 喜欢，互惠 |
| 指导/技术 | 中等权威 + 团结 | 重权威 |
| 协作 | 团结 + 承诺 | 权威，喜欢 |
| 参考 | 仅清晰 | 所有说服 |

## 为什么有效：心理学

**明亮线规则减少合理化：**

- "YOU MUST" 移除决策疲劳
- 绝对语言消除"这是例外吗？"问题
- 明确反合理化对抗特定漏洞

**实施意图创建自动行为：**

- 清晰触发 + 必需行动 = 自动执行
- "当 X 时，做 Y" 比"一般做 Y" 更有效
- 减少合规认知负荷

**LLM 是类人：**

- 在包含这些模式的文本上训练
- 训练数据中权威语言先于合规
- 承诺序列（声明 → 行动）频繁建模
- 社会证明模式（每个人都做 X）建立规范

## 伦理使用

**合法：**

- 确保关键实践被遵循
- 创建有效文档
- 防止可预测失败

**非法：**

- 为个人利益操纵
- 创建虚假紧迫性
- 基于内疚的合规

**测试：** 如果用户完全理解，这个技术是否会服务于他们的真正利益？

## 快速参考

设计提示时，问：

1. **它是什么类型？**（纪律 vs. 指导 vs. 参考）
2. **我试图改变什么行为？**
3. **哪个原则适用？**（通常纪律的权威 + 承诺）
4. **我组合太多吗？**（不要使用全部七个）
5. **这是伦理的吗？**（服务用户的真正利益？）
