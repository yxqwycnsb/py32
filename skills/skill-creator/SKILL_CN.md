# 技能创建者

一个用于创建新技能并迭代改进它们的技能。

从高层次来看，创建技能的过程如下：

- 决定你想要技能做什么以及大致如何做
- 写一个技能草稿
- 创建几个测试提示，并在有访问技能的Claude上运行它们
- 帮助用户评估结果，既定性和定量
  - 当运行在后台进行时，如果没有评估（如果有，你可以按原样使用或修改），起草一些定量评估。然后向用户解释它们（或如果已经存在，解释现有的）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果，让他们查看，并让他们查看定量指标
- 根据用户对结果评估的反馈重写技能（以及如果从定量基准中出现任何明显缺陷）
- 重复直到满意
- 扩展测试集并在更大规模上再次尝试

当使用这个技能时，你的工作是找出用户在这个过程中的位置，然后跳入并帮助他们通过这些阶段。例如，也许他们说“我想为X创建一个技能”。你可以帮助缩小他们的意思，写一个草稿，写测试案例，找出他们想要如何评估，运行所有提示，并重复。

另一方面，也许他们已经有一个技能草稿。在这种情况下，你可以直接进入评估/迭代循环。

当然，你应该总是灵活的，如果用户说“我不需要运行一堆评估，只需与我共鸣”，你可以那样做。

然后在技能完成后（但再次，顺序是灵活的），你也可以运行技能描述改进器，我们有一个单独的脚本用于此，以优化技能的触发。

酷吗？酷。

## 与用户沟通

技能创建者可能被各种熟悉编码术语的人使用。如果你没有听说过（你怎么可能听说，这是最近才开始的趋势），现在有一个趋势，Claude的力量激励水管工打开他们的终端，父母和祖父母谷歌“如何安装npm”。另一方面，大多数用户可能相当计算机素养。

所以请注意上下文线索来理解如何表达你的沟通！在默认情况下，只是给你一些想法：

- “evaluation”和“benchmark”是边界的，但OK
- 对于“JSON”和“assertion”，你想看到用户知道这些东西的严重线索，然后在使用它们之前解释它们

如果不确定，可以简要解释术语，并随时澄清术语，如果不确定用户是否会理解。

## 创建技能

### 捕获意图

从理解用户的意图开始。当前对话可能已经包含用户想要捕获的工作流程（例如，他们说“把这个变成技能”）。如果是这样，首先从对话历史中提取答案——使用的工具、步骤序列、用户做的修正、观察到的输入/输出格式。用户可能需要填补空白，并在继续下一步之前确认。

1. 这个技能应该让Claude做什么？
2. 什么时候应该触发？（什么用户短语/上下文）
3. 预期的输出格式是什么？
4. 我们应该设置测试案例来验证技能工作吗？具有客观可验证输出的技能（文件变换、数据提取、代码生成、固定工作流程步骤）受益于测试案例。具有主观输出的技能（写作风格、艺术）通常不需要它们。根据技能类型建议适当的默认，但让用户决定。

### 采访和研究

主动询问边缘案例、输入/输出格式、示例文件、成功标准和依赖项。等到你整理好这个部分后再写测试提示。

检查可用的MCP——如果对研究有用（搜索文档、查找类似技能、查找最佳实践），如果可用，通过子代理并行研究，否则内联。准备好上下文以减少用户的负担。

### 写SKILL.md

基于用户采访，填写这些组件：

- **name**：技能标识符
- **description**：何时触发，它做什么。这是主要的触发机制——包括技能做什么以及何时使用的具体上下文。所有“何时使用”信息都放在这里，而不是正文中。注意：目前Claude有倾向于“undertrigger”技能——在有用时不使用它们。为了对抗这个，请让技能描述有点“pushy”。例如，不是“How to build a simple fast dashboard to display internal Anthropic data.”，你可能会写“How to build a simple fast dashboard to display internal Anthropic data. Make sure to use this skill whenever the user mentions dashboards, data visualization, internal metrics, or wants to display any kind of company data, even if they don't explicitly ask for a 'dashboard.'”
- **compatibility**：所需工具、依赖项（可选，很少需要）
- **其余的技能：）**

### 技能写作指南

#### 技能的解剖

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (name, description 必需)
│   └── Markdown 指令
└── 捆绑资源 (可选)
    ├── scripts/    - 可执行代码用于确定性/重复任务
    ├── references/ - 文档按需加载到上下文中
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进披露

技能使用三层加载系统：
1. **元数据** (name + description) - 总是上下文（~100字）
2. **SKILL.md 正文** - 每当技能触发时上下文（<500行理想）
3. **捆绑资源** - 按需（无限，脚本可以执行而不加载）

这些字数是近似的，如果你需要，可以随意更长。

**关键模式：**
- 保持SKILL.md低于500行；如果你接近这个限制，添加额外层次以及清晰指针，关于使用技能的模型应该去哪里下一步跟随。
- 从SKILL.md清晰引用文件，并指导何时读取它们
- 对于大参考文件（>300行），包括目录

**域组织**：当技能支持多个域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md (工作流程 + 选择)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude只读取相关的参考文件。

#### 缺乏惊喜原则

这不用说，但技能不得包含恶意软件、利用代码，或任何可能损害系统安全的内容。技能的内容如果描述，不应该在意图上让用户惊讶。不要同意创建误导技能或设计用于促进未经授权访问、数据泄露或其他恶意活动的技能。不过，像“roleplay as an XYZ”之类的是OK的。

#### 写作模式

在指令中偏好使用祈使形式。

**定义输出格式** - 你可以这样做：
```markdown
## 报告结构
总是使用这个确切模板：
# [标题]
## 执行摘要
## 关键发现
## 推荐
```

**示例模式** - 包括示例是有用的。你可以这样格式化它们（但如果“Input”和“Output”在示例中，你可能想稍微偏离）：
```markdown
## 提交消息格式
**示例 1：**
Input: Added user authentication with JWT tokens
Output: feat(auth): implement JWT-based authentication
```

### 写作风格

尝试向模型解释为什么事情重要，而不是沉重的MUST。使用理论思维，并尝试使技能一般而不是超级狭窄到特定示例。从写草稿开始，然后用新鲜眼光看它并改进。

### 测试案例

写技能草稿后，想出2-3个现实的测试提示——真实用户会说的那种东西。与用户分享它们：[你不必使用这个确切语言] “这里是我想尝试的几个测试案例。这些看起来对吗，或者你想添加更多？”然后运行它们。

将测试案例保存到 `evals/evals.json`。不要写断言——只是提示。你将在下一步起草断言，同时运行进行中。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "User's task prompt",
      "expected_output": "Description of expected result",
      "files": []
    }
  ]
}
```

见 `references/schemas.md` 获取完整模式（包括 `assertions` 字段，你稍后会添加）。

## 运行和评估测试案例

这个部分是一个连续序列——不要中途停止。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 作为技能目录的兄弟。在工作区内，按迭代组织结果（`iteration-1/`、`iteration-2/` 等），在其中，每个测试案例得到一个目录（`eval-0/`、`eval-1/` 等）。不要提前创建所有这些——只是随着进行创建目录。

### 步骤1：同时生成所有运行（有技能和基准）

对于每个测试案例，同时生成两个子代理——一个有技能，一个没有。这很重要：不要先生成有技能运行，然后回来为基准。一次启动一切，这样一切都大约同时完成。

**有技能运行：**

```
执行这个任务：
- 技能路径： <path-to-skill>
- 任务： <eval prompt>
- 输入文件： <eval files if any, or "none">
- 保存输出到： <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 要保存的输出： <用户关心的——例如，“.docx文件”，“最终CSV”>
```

**基准运行**（相同提示，但基准取决于上下文）：
- **创建新技能**：根本没有技能。相同提示，没有技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：旧版本。在编辑之前，拍摄技能快照（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后指向基准子代理快照。保存到 `old_skill/outputs/`。

为每个测试案例写一个 `eval_metadata.json`（断言现在可以为空）。给每个评估一个基于测试内容的描述性名称——不仅仅是“eval-0”。也用这个名称作为目录。如果这个迭代使用新的或修改的评估提示，为每个新评估目录创建这些文件——不要假设它们从以前迭代延续。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "The user's task prompt",
  "assertions": []
}
```

### 步骤2：当运行进行时，起草断言

不要只是等待运行完成——你可以用这个时间生产性地使用。起草每个测试案例的定量断言，并向用户解释它们。如果断言已经在 `evals/evals.json` 中存在，审查它们并解释它们检查什么。

好的断言是客观可验证的，并有描述性名称——它们应该在基准查看器中清晰读取，这样有人瞥一眼结果立即理解每个检查什么。主观技能（写作风格、设计质量）更好地定性评估——不要强迫断言到需要人类判断的事情上。

一旦起草，更新 `eval_metadata.json` 文件和 `evals/evals.json` 与断言。也向用户解释他们将在查看器中看到什么——定性输出和定量基准。

### 步骤3：当运行完成时，捕获时间数据

当每个子代理任务完成时，你收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将这个数据保存到运行目录的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获这个数据的唯一机会——它通过任务通知来，不是持久化在其他地方。每个通知到达时处理，而不是尝试批处理。

### 步骤4：评分、聚合并启动查看器

一旦所有运行完成：

1. **评分每个运行**——生成评分子代理（或内联评分）读取 `agents/grader.md` 并根据输出评估每个断言。将结果保存到每个运行目录的 `grading.json`。grading.json 期望数组必须使用字段 `text`、`passed` 和 `evidence`（不是 `name`/`met`/`details` 或其他变体）——查看器依赖这些确切字段名称。对于可以编程检查的断言，写并运行脚本而不是眼球——脚本更快、更可靠，并可以在迭代中重用。

2. **聚合到基准**——从技能创建者目录运行聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   这产生 `benchmark.json` 和 `benchmark.md` 与每个配置的通过率、时间和令牌，具有均值 ± 标准差和增量。如果手动生成 benchmark.json，见 `references/schemas.md` 获取查看器期望的确切模式。
将每个 with_skill 版本放在其基准对应物之前。

3. **做分析师通过**——读取基准数据并表面聚合统计可能隐藏的模式。见 `agents/analyzer.md`（“分析基准结果”部分）查找什么——像总是通过无论技能的断言（非歧视）、高方差评估（可能不稳定），以及时间/令牌权衡。

4. **启动查看器** 与定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代2+，也传递 `--previous-workspace <workspace>/iteration-<N-1>`。

   **Cowork / 无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示，使用 `--static <output_path>` 写独立HTML文件而不是启动服务器。当用户点击“提交所有审查”时，反馈将作为 `feedback.json` 文件下载。下载后，将 `feedback.json` 复制到工作区目录以供下一次迭代拾取。

注意：请使用 generate_review.py 创建查看器；没有必要写自定义HTML。

5. **告诉用户** 像这样：“我已经在你的浏览器中打开了结果。有两个标签——‘输出’让你点击每个测试案例并留下反馈，‘基准’显示定量比较。当你完成时，回这里告诉我。”

### 用户在查看器中看到什么

“输出”标签一次显示一个测试案例：
- **提示**：给定的任务
- **输出**：技能产生的文件，如果可能内联渲染
- **以前输出**（迭代2+）：折叠部分显示上次迭代的输出
- **正式评分**（如果运行了评分）：折叠部分显示断言通过/失败
- **反馈**：自动保存他们输入的文本框
- **以前反馈**（迭代2+）：他们上次评论，显示在文本框下面

“基准”标签显示统计摘要：每个配置的通过率、时间和令牌使用，具有每评估细分和分析师观察。

导航通过上一个/下一个按钮或箭头键。当完成时，他们点击“提交所有审查”保存所有反馈到 `feedback.json`。

### 步骤5：读取反馈

当用户告诉你他们完成时，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "the chart is missing axis labels", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "perfect, love this", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为它很好。专注于用户有具体抱怨的测试案例改进。

当你完成时杀死查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已经运行了测试案例，用户已经审查了结果，现在你需要基于他们的反馈使技能更好。

### 如何思考改进

1. **从反馈中概括。** 这里发生的大图是我们在尝试创建可以百万次使用的技能（也许字面，也许甚至更多谁知道）跨越许多不同提示。这里你和用户在只有几个示例上迭代一遍又一遍，因为它帮助更快移动。用户知道这些示例里里外外，对他们来说快速评估新输出。但如果你们共同开发的技能只为那些示例工作，它是无用的。与其放入琐碎的过度拟合变化，或压迫性约束的MUST，如果有一些顽固问题，你可能尝试分支并使用不同隐喻，或推荐不同工作模式。尝试相对便宜，也许你会落地一些伟大的东西。

2. **保持提示精简。** 移除没有拉动重量的东西。确保读取转录，不是只是最终输出——如果看起来技能在浪费时间做不生产性的事情，你可以尝试摆脱使它那样做的技能部分，看看会发生什么。

3. **解释为什么。** 努力解释一切你要求模型做的**为什么**。今天的LLM是*聪明的*。它们有好的理论思维，当给一个好工具时可以超越死记硬背指令并真正使事情发生。即使用户的反馈是简短或沮丧的，尝试实际理解任务和为什么用户写他们写的，以及他们实际写了什么，然后将这个理解传输到指令中。如果你发现自己在写ALL CAPS的ALWAYS或NEVER，或使用超级刚性结构，那是黄旗——如果可能，重构并解释推理，这样模型理解为什么你要求的那个东西重要。那是一个更人性化、强大和有效的途径。

4. **寻找跨测试案例的重复工作。** 读取测试运行的转录，并注意如果所有子代理都独立写了类似助手脚本或对某事采取相同多步骤方法。如果所有3个测试案例导致子代理写 `create_docx.py` 或 `build_chart.py`，那是强信号技能应该捆绑那个脚本。写一次，放在 `scripts/`，告诉技能使用它。这节省每个未来调用重新发明轮子。

这个任务相当重要（我们在这里尝试创建每年数十亿美元的经济价值！）你的思考时间不是阻塞；花时间真正思考。我建议写一个修订草稿，然后重新看它并改进。真正尽力进入用户的头脑，理解他们想要和需要什么。

### 迭代循环

改进技能后：

1. 将你的改进应用到技能
2. 将所有测试案例重新运行到新的 `iteration-<N+1>/` 目录，包括基准运行。如果你正在创建新技能，基准总是 `without_skill`（无技能）——那在迭代中保持相同。如果你正在改进现有技能，使用你的判断什么作为基准有意义：用户进来的原始版本，或以前迭代。
3. 用 `--previous-workspace` 指向以前迭代启动审查者
4. 等待用户审查并告诉你他们完成
5. 读取新反馈，再次改进，重复

继续直到：
- 用户说他们满意
- 反馈都是空的（一切看起来好）
- 你没有做出有意义进展

## 高级：盲比较

对于你想要两个技能版本之间更严格比较的情况（例如，用户问“新版本实际上更好吗？”），有一个盲比较系统。读取 `agents/comparator.md` 和 `agents/analyzer.md` 获取细节。基本想法是：给两个输出到一个独立代理而不告诉它哪个是哪个，让它判断质量。然后分析为什么赢家赢。

这是可选的，需要子代理，大多数用户不需要。人类审查循环通常足够。

## 描述优化

SKILL.md frontmatter 中的描述字段是决定Claude是否调用技能的主要机制。创建或改进技能后，提供优化描述以提高触发准确性。

### 步骤1：生成触发评估查询

创建20个评估查询——混合应该触发和不应该触发。保存为JSON：

```json
[
  {"query": "the user prompt", "should_trigger": true},
  {"query": "another prompt", "should_trigger": false}
]
```

查询必须是现实的，Claude Code 或 Claude.ai 用户会实际输入的。不是抽象请求，而是具体的、有细节的请求。文件路径、用户工作或情况的个人上下文、列名和值、公司名、URL。有点背景故事。有些可能小写或包含缩写或拼写错误或随意演讲。使用不同长度混合，专注于边缘案例而不是使它们清晰（用户会得到签署机会）。

坏：“Format this data”，“Extract text from PDF”，“Create a chart”

好：“ok so my boss just sent me this xlsx file (its in my downloads, called something like 'Q4 sales final FINAL v2.xlsx') and she wants me to add a column that shows the profit margin as a percentage. The revenue is in column C and costs are in column D i think”

对于**应该触发**查询（8-10），想想覆盖。你想要相同意图的不同措辞——一些正式，一些随意。包括用户没有明确命名技能或文件类型但清楚需要它的案例。扔一些不常见用例和这个技能与另一个竞争但应该赢的案例。

对于**不应该触发**查询（8-10），最有价值的是近失——与技能共享关键词或概念但实际需要不同的查询。想想相邻域、模糊措辞哪里天真关键词匹配会触发但不应该，和查询触及技能做的事情但在另一个工具更合适的上下文。

关键避免：不要使不应该触发查询明显无关。“Write a fibonacci function”作为PDF技能的负面测试太容易——它不测试任何东西。负面案例应该真正棘手。

### 步骤2：与用户审查

使用HTML模板向用户呈现评估集：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的JSON数组（没有引号——它是JS变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能的名字
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能的当前描述
3. 写到临时文件（例如，`/tmp/eval_review_<skill-name>.html`）并打开它：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换应该触发、添加/移除条目，然后点击“导出评估集”
5. 文件下载到 `~/Downloads/eval_set.json`——检查下载文件夹的最新版本，以防有多个（例如，`eval_set (1).json`）

这个步骤重要——坏评估查询导致坏描述。

### 步骤3：运行优化循环

告诉用户：“这将需要一些时间——我将在后台运行优化循环并定期检查。”

将评估集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用你的系统提示中的模型ID（驱动当前会话的那个）这样触发测试匹配用户实际体验。

当它运行时，定期尾随输出给用户更新它在哪个迭代和分数看起来如何。

这自动处理完整优化循环。它将评估集分成60%训练和40%保留测试，评估当前描述（运行每个查询3次获取可靠触发率），然后用扩展思考调用Claude基于失败提出改进。它重新评估每个新描述在训练和测试上，迭代最多5次。当完成时，它在浏览器中打开HTML报告显示每迭代结果并返回JSON与 `best_description`——由测试分数选择而不是训练分数以避免过度拟合。

### 技能触发如何工作

理解触发机制帮助设计更好评估查询。技能出现在Claude的 `available_skills` 列表与他们的名字 + 描述，Claude基于那个描述决定是否咨询技能。重要知道的是Claude只为它不能容易处理的自己任务咨询技能——像“read this PDF”这样的简单、一步查询可能不触发技能即使描述完美匹配，因为Claude可以用基本工具直接处理它们。复杂、多步或专门查询可靠触发技能当描述匹配时。

这意味着你的评估查询应该实质足够Claude会实际受益从咨询技能。像“read file X”这样的简单查询是坏测试案例——它们不会触发技能无论描述质量如何。

### 步骤4：应用结果

从JSON输出取 `best_description` 并更新技能的SKILL.md frontmatter。向用户展示前后并报告分数。

## Claude.ai特定指令

在Claude.ai中，核心工作流程相同（草稿 → 测试 → 审查 → 改进 → 重复），但因为Claude.ai没有子代理，一些机制改变。这里是什么适应：

**运行测试案例**：没有子代理意味着没有并行执行。对于每个测试案例，读取技能的SKILL.md，然后跟随其指令自己完成测试提示。一次一个。这不如独立子代理严格（你写了技能，你也在运行它，所以你有完整上下文），但它是有用理智检查——人类审查步骤补偿。跳过基准运行——只是用技能完成请求的任务。

**审查结果**：如果你不能打开浏览器（例如，Claude.ai的VM没有显示，或你在远程服务器），完全跳过浏览器审查者。相反，直接在对话中呈现结果。对于每个测试案例，显示提示和输出。如果输出是用户需要看到的文件（像.docx或.xlsx），保存到文件系统并告诉他们在哪里这样他们可以下载并检查。内联询问反馈：“这看起来如何？任何你想改变的？”

**基准**：跳过定量基准——它依赖基准比较没有子代理没有意义。专注于用户的定性反馈。

**迭代循环**：相同之前——改进技能，重新运行测试案例，询问反馈——只是中间没有浏览器审查者。你仍然可以将结果组织到文件系统上的迭代目录如果你有。

**描述优化**：这个部分需要 `claude` CLI工具（具体 `claude -p`）只在Claude Code中可用。如果你在Claude.ai，跳过它。

**盲比较**：需要子代理。跳过它。

**打包**：`package_skill.py` 脚本在任何有Python和文件系统的地方工作。在Claude.ai，你可以运行它，用户可以下载结果 `.skill` 文件。

## Cowork特定指令

如果你在Cowork，主要知道的是：

- 你有子代理，所以主要工作流程（并行生成测试案例，运行基准，评分等）都工作。（然而，如果你遇到严重超时问题，OK串行运行测试提示而不是并行。）
- 你没有浏览器或显示，所以当生成评估查看器时，使用 `--static <output_path>` 写独立HTML文件而不是启动服务器。然后提供用户可以点击打开浏览器中HTML的链接。
- 出于某种原因，Cowork设置似乎使Claude不愿意在运行测试后生成评估查看器，所以只是重申：无论你在Cowork还是Claude Code，在运行测试后，你应该总是生成评估查看器供人类在你自己评估输入并尝试修正之前查看示例，使用 `generate_review.py`（不是写你自己的精品html代码）。抱歉提前但我要大写这里：*在*你自己评估输入之前生成评估查看器。你想要尽快把它们放在人类面前！
- 反馈工作不同：因为没有运行服务器，查看器的“提交所有审查”按钮将下载 `feedback.json` 作为文件。你可以从那里读取它（你可能需要先请求访问）。
- 打包工作——`package_skill.py` 只需Python和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）应该在Cowork中工作很好因为它通过子进程使用 `claude -p`，不是浏览器，但请保存直到你完全完成技能并用户同意它在好形状。
